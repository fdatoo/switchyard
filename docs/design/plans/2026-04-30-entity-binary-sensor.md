# Entity Binary-Sensor Proto Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Implement the proto changes specified in `docs/design/specs/2026-04-30-entity-binary-sensor-design.md`: rename `Sensor` → `NumericSensor`, add a new `BinarySensor` variant to `Attributes.kind`, and update the daemon's only consumer.

**Architecture:** This is a refactor + additive proto change, not a feature. There is no new behaviour to test-drive — the proto compiler and the existing test suite are the validators. The change is wire-compatible (field number 12 is preserved), so the failure mode is purely Go-source breakage which the build catches deterministically.

The work is laid out in two tasks. Task 1 edits the proto and regenerates, leaving the tree intentionally non-compiling on `internal/daemon/daemon.go`. Task 2 fixes the daemon, runs the full build + test suite, and commits the whole change as a single atomic commit. There is **no intermediate commit** between the two tasks because a half-applied rename leaves `main` non-compiling; we never push that state.

**Tech Stack:** protobuf v3 + `buf` (configured via `buf.gen.yaml`, `buf.yaml`); `task proto` runs `buf generate`. Go 1.25. No new external dependencies.

- [ ] **Step 1: Edit the proto file**

Replace the contents of `proto/gohome/entity/v1/attributes.proto` with:

```proto
// See docs/proto-hygiene.md for grouping conventions.

syntax = "proto3";

package gohome.entity.v1;

// Attributes carries both capabilities (static) and live state (dynamic)
// for an entity. Specific payloads land in later milestones — C1 only
// needs the envelope compiling. Future oneof variants get added here.
message Attributes {
  // Old name retired in favor of `numeric_sensor` for symmetry with
  // `binary_sensor`. Field number 12 stays (wire-compatible). Per
  // dev/proto-hygiene.md rule 3, we reserve the old name to prevent
  // accidental reuse.
  reserved "sensor";

  oneof kind {
    // 10-19: actuator/sensor domains
    Light          light           = 10;
    Switch         switch_device   = 11;
    NumericSensor  numeric_sensor  = 12;  // was: Sensor sensor; field number stays
    BinarySensor   binary_sensor   = 13;
  }
  // 90-99: cross-cutting metadata
  // available is the driver's claim that the entity is currently reachable.
  // Default false (zero) means unknown / unreachable; drivers explicitly set
  // true after confirming connectivity. The state cache surfaces this.
  bool available = 90;
}

message Light {
  bool   on         = 1;
  uint32 brightness = 2;   // 0-255
  uint32 color_temp = 3;   // mireds; 0 if unsupported
  uint32 color_rgb  = 4;   // 0xRRGGBB; 0 if unsupported / not active
  // RGB/XY/capabilities flesh out later; C1 only needs the message existing.
}

message Switch {
  bool on = 1;
}

message NumericSensor {
  string unit  = 1;
  double value = 2;
}

message BinarySensor {
  // on encodes the active reading: motion detected, contact closed,
  // water present, etc. Drivers may invert at the source if a device
  // reports the inverse polarity natively (the inversion choice is the
  // driver's, not the proto's).
  bool on = 1;
}
```

Notes for the executor:
- The `reserved "sensor";` statement reserves the **field name** only. Field number 12 is **not** reserved because we are reusing it for `numeric_sensor`. Both reserving the name and reusing the number are valid in proto3.
- `Sensor` is renamed to `NumericSensor` (message type rename) and the oneof field name changes from `sensor` to `numeric_sensor`. The Go API consequence is that `entityv1.Sensor` becomes `entityv1.NumericSensor`, the oneof variant tag `*entityv1.Attributes_Sensor` becomes `*entityv1.Attributes_NumericSensor`, and the inner field on the variant is `NumericSensor` instead of `Sensor`.
