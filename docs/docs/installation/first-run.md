# First run

This page applies to all install methods. After you have installed switchyard via the [static binary](binary.md), [Docker](docker.md), or [systemd / packages](systemd.md) route, follow these steps to create a config, validate it, and confirm the daemon is running.

## 1. Create the config directory

=== "Linux (bare metal / systemd)"

    The system-wide convention places config in `/etc/switchyard/`:

    ```bash
    sudo mkdir -p /etc/switchyard
    sudo chown "$(id -u):$(id -g)" /etc/switchyard  # if running as your own user
    ```

    If you prefer a user-local install, use `~/.config/switchyard/` instead:

    ```bash
    mkdir -p ~/.config/switchyard
    ```

=== "Docker"

    Create the directories that will be bind-mounted into the container:

    ```bash
    mkdir -p ./config ./data
    ```

    When you run the container the `./config` directory maps to `/config` inside it. All commands in this section that reference a path refer to paths *inside the container*; your actual files live in `./config/` on the host.

=== "macOS"

    ```bash
    mkdir -p ~/.config/switchyard
    ```

## 2. Generate a starter config

Run `switchyard config init` to write a minimal `main.pkl` to the config directory:

```bash
# Bare metal
switchyard config init --config-dir ~/.config/switchyard

# Docker (runs the CLI inside the running container)
docker exec switchyardd switchyard config init --config-dir /config
```

This creates `main.pkl` and a `lib/` directory with helper modules. The generated file looks like:

```pkl
/// switchyard configuration — generated by `switchyard config init`
amends "switchyard:base.pkl"

/// Your home's display name.
name = "My Home"

/// Areas (rooms, floors, outdoor zones).
/// See the configuration docs to add areas, entities, and drivers.
areas {}

/// Driver instances.
/// Each entry loads a driver binary and passes it typed config.
drivers {}

/// Automations.
automations {}
```

If you prefer to write the file by hand, this is the absolute minimum that passes validation:

```pkl
amends "switchyard:base.pkl"

name = "My Home"
areas {}
drivers {}
automations {}
```

Save this as `main.pkl` in your config directory.

## 3. Validate the config

You can validate the config either before or after starting the daemon:

**Before the daemon starts** — use `switchyardd --validate-only`. This parses the config and exits without running a persistent daemon process:

```bash
# Bare metal
switchyardd --validate-only --config ~/.config/switchyard/main.pkl
```

**After the daemon starts** — use the CLI, which connects to the running daemon:

```bash
# Bare metal
switchyard config validate

# Specify a path explicitly
switchyard config validate --config ~/.config/switchyard/main.pkl

# Docker (daemon must already be running)
docker exec switchyardd switchyard config validate
```

A clean config prints:

```
Config OK (0 warnings, 0 errors)
  root: ~/.config/switchyard/main.pkl
  areas: 0
  drivers: 0
  automations: 0
```

Fix any errors before proceeding. Common issues on a fresh config are a missing `amends` directive or a typo in a Pkl module import. The error messages include the file path and line number.

## 4. Start the daemon

=== "Bare metal (foreground, for testing)"

    ```bash
    switchyardd --config ~/.config/switchyard/main.pkl \
             --data-dir ~/.local/share/switchyard
    ```

    The daemon logs to stdout. Press `Ctrl-C` to stop.

=== "systemd"

    ```bash
    sudo systemctl start switchyardd
    sudo systemctl status switchyardd
    ```

=== "Docker Compose"

    ```bash
    docker compose up -d
    docker compose logs -f switchyardd
    ```

=== "Docker (single container)"

    ```bash
    docker run -d \
      --name switchyardd \
      --restart unless-stopped \
      -p 8080:8080 \
      -p 9090:9090 \
      -v "$(pwd)/config:/config" \
      -v "$(pwd)/data:/data" \
      ghcr.io/fynn-labs/switchyard:latest
    ```

## 5. Confirm the daemon is healthy

```bash
switchyard status
```

If the daemon is running, this returns:

```
switchyardd v0.1.0 — healthy
  uptime:   12s
  entities: 0 registered
  drivers:  0 loaded
  events:   1 (last: SystemStarted 12s ago)
  api:      http://localhost:8080
  mcp:      http://localhost:9090
```

If the CLI cannot reach the daemon:

```bash
# Set the server URL explicitly if it is not on localhost:8080
switchyard --server http://<your-host>:8080 status

# Or export it for the session
export GOHOME_SERVER=http://<your-host>:8080
switchyard status
```

## 6. Open the web UI

Navigate to `http://localhost:8080` (or `http://<your-host>:8080`) in a browser. You should see the switchyard dashboard. At this point it will be empty — no entities, no areas, no drivers — because the config is minimal. That is expected.

## 7. Create your first API token (optional, recommended)

By default, `switchyardd` creates a local admin session for the user that started it. For programmatic access (the CLI from another machine, an AI agent, scripts), create an API token:

```bash
switchyard auth token create --name "my-laptop" --role admin
```

Store the token securely. Pass it with `--token <token>` or set `GOHOME_TOKEN=<token>` in your environment.

## What to do next

A minimal config without areas, entities, or drivers is a running but empty daemon. The next steps are:

- Read [Concepts](../concepts/index.md) to understand how switchyard models entities, areas, drivers, and events before writing config
- Follow [Configuration](../configuration/index.md) to add areas and entities to your config
- Install a driver — see [Drivers](../drivers/index.md)
- (If migrating from Home Assistant) See the [Migration guide](../migration/index.md)

## Troubleshooting

**`switchyard status` says "connection refused"**

The daemon is not running, or it is listening on a different address. Check `systemctl status switchyardd` or `docker ps`. Confirm the port with `--server http://<host>:<port> status`.

**`switchyard config validate` fails with "module not found"**

The Pkl `switchyard:*` modules are embedded in `switchyardd`. When using `switchyard config validate`, the CLI connects to the running daemon (which knows the embedded module versions) — so the daemon must be running. To validate before starting the daemon, use `switchyardd --validate-only --config <path>` instead.

**Permission denied on `/etc/switchyard` or `/var/lib/switchyard`**

If you installed via the systemd unit and created a `switchyardd` system user, run config-directory operations as that user:

```bash
sudo -u switchyardd switchyard config validate --config /etc/switchyard/main.pkl
```
